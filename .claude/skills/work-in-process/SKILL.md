# WorkInProcess

围绕 `WIP.md` 文件，生成并维护可执行的技术方案，覆盖从需求设计、自查、编码到复核的完整流程。

## 执行

若调用时未指定子命令，直接返回以下介绍信息：

可用子命令：

| 命令 | 功能 |
|------|------|
| `wip` | 展示技能简介及可用子命令 |
| `wip-init` | 创建或重置 WIP.md，写入需求骨架 |
| `wip-build` | 从对话中提取需求，完善 WIP.md（生成设计方案 + 执行计划） |
| `wip-check` | 编码前快速自查——校验方案自洽性、步骤依赖、状态机/数据模型一致性 |
| `wip-code` | 按 WIP.md 执行计划逐步骤编写代码 |
| `wip-review` | 编码后全面复核——对照源码逐步骤检查实现是否正确、有无遗漏或引入问题 |
| `wip-feishu-upload` | 将 WIP.md 上传为飞书云文档，按项目归类到指定文件夹 |
| `wip-feishu-list` | 列出飞书云空间中已上传的 WIP 文档列表（可选 page_size page，默认 100 / 第1页） |
| `wip-feishu-delete` | 删除飞书云空间中的 WIP 文档（按关键字/doc_id/--all） |
| `wip-feishu-search` | 按标题关键字搜索已上传的 WIP 文档 |
| `wip-feishu-read` | 按关键字搜索并阅读所有匹配的 WIP 文档内容 |

工作流程：`wip-init` → `wip-build` → `wip-check` → `wip-code` → `wip-review` → `wip-feishu-upload`（可选）
飞书管理：`wip-feishu-list` | `wip-feishu-search` | `wip-feishu-read` | `wip-feishu-delete`

## 子命令

### `wip-init`

创建或重置 `WIP.md`。

1. 创建 `WIP.md`（若不存在），写入骨架：
   ```
   # WIP: {需求标题}
   ## 参考资料
   ## 需求背景
   ## 设计方案
   ## 执行计划
   ## 变更文件汇总
   ## 构建顺序
   ```
2. 将 `WIP.md` 加入 `.gitignore`（若尚未忽略）——WIP.md 是个人临时工作文档，不入库、不参与团队协作，用完即弃
3. 若 `WIP.md` 已存在，清空内容后写入骨架

### `wip-build`

从对话中提取需求，完善 `WIP.md`（生成「设计方案」与「执行计划」）。

1. 读取当前项目代码结构，理解技术栈、命名风格和现有模式。
2. 从对话中提取所有需求要点，写入「需求背景」——包括：触发条件、范围边界、前置校验、状态约束、幂等语义等。
3. 生成「设计方案」——面向临时评审的总体设计概述：
   - **总体思路**：用 3-4 句话讲清楚「要解决什么问题 → 用什么方式解决 → 核心设计原则是什么」。不说具体表名、字段名、API 路径，让非技术读者也能看懂。
   - **架构概要**：说明改动涉及的系统层次（数据层/业务层/接口层）、各层职责划分、模块间调用关系，顺带标注涉及的数据表、接口、上下游依赖及是否向后兼容。
   - **关键决策**：用表格列出 2-5 个关键设计取舍 `# | 决策 | 方案 | 理由 | 风险`（如：为何选择该方案而非其他、状态机流转设计依据）。
   - 整体控制在 1 页篇幅，语言精炼，便于快速评审决策。

4. 按依赖顺序生成「执行计划」——这是核心产出：
   - 将需求按层次分阶段（数据层 → 模型/实体 → 业务逻辑 → 接口/入口 → 联调验证），每阶段拆分为若干**可执行步骤**。
   - 每个步骤用统一模板呈现（参考「通用规则·每个步骤必须可执行」）：`文件` / `位置`（精确到方法名+行号锚点）/ `背景`（为何改）/ `操作`（具体代码/SQL）/ `验证`（编译/单测/联调）。
   - 联调验证阶段用测试矩阵呈现：`# | 场景 | 前置条件 | 预期结果`，覆盖正常流、边界、幂等、并发等场景。
   - 步骤号一经分配不复用；迭代中被取消的步骤保留占位并注明「已移除 + 原因」，不静默删除，保持可追溯。
5. 生成「变更文件汇总」——独立章节，用表格 `# | 文件 | 操作 | 对应阶段` 列出每个文件的操作类型（新增/修改/删除）及对应执行步骤，作为回溯速查表。
6. 生成「构建顺序」——用依赖图标注阶段间推进顺序及阶段内并行链路（如转移线/列表线/明细线），点明关键步骤的前置依赖。
7. 补齐「参考资料」中必要的项目上下文（业务流程图、数据表关系、关键方法签名等），关键方法标注 `文件:行号` 锚点，供后续步骤引用。

### `wip-check`

编码前轻量自查——校验方案自洽性，不深入比对源码（源码级精确校验留给 `wip-review`）。

1. **需求覆盖检查**：
   - 需求背景中的每条要点在执行计划中是否有对应步骤？
   - 范围边界是否清晰？（什么做、什么明确不做）
   - 「设计方案」与需求目标是否一致？
2. **方案自洽性检查**：
   - 步骤依赖是否满足？（数据层先于业务层，定义先于引用）
   - 状态机流转是否完整？是否有不可达或死锁状态？
   - 步骤间数据传递是否自洽（上一步产出是否为下一步所需输入）？
3. **数据模型一致性**：
   - 新增字段/实体命名是否与已有定义冲突？
   - 字段类型是否符合项目既有风格和语言惯例？
   - 索引/约束是否覆盖关键查询条件？
4. **格式完整性**：
   - 每个步骤是否齐备「文件/位置/操作/验证」要素？位置是否精确到方法名+行号？
   - 「变更文件汇总」「构建顺序」两个独立章节是否存在且与执行计划一致？
   - 联调验证阶段是否有测试矩阵，覆盖边界/幂等场景？
5. 输出问题清单，修正 WIP.md 后重新确认。
6. 通过标准：无阻塞性问题，可进入编码。

### `wip-code`

按 `WIP.md` 执行计划逐步骤编写代码。

1. 读取 WIP.md，确认 `wip-check` 已通过。
2. 用 `git status` 确认工作区干净（或记录已知初始状态），保证后续 `wip-review` 的 diff 对比只反映本次改动。
3. 按步骤顺序依次实现：
   - 打开对应文件，读取当前内容。
   - 执行代码修改（新增/修改/删除），精确按 WIP.md 中的代码/SQL 实施。
   - 若发现 WIP.md 与实际代码有出入（字段缺失、签名不符、方案有疏漏等），先回写更新 WIP.md 再继续，保持文档与代码同步。
   - 每步完成后立即验证（编译/运行单测），标记进度。
4. 全部步骤完成后执行项目构建验证。
5. 若某步骤失败，暂停并报告问题，等待人工介入。

### `wip-review`

编码后全面复核——对照已完成源码，逐步骤检查实现是否正确、有无遗漏或引入问题。

1. **逐步骤对照复核**——每个步骤对照 WIP.md 与已修改的源码：
   - 代码逻辑是否与 WIP.md 描述一致？有无实现偏差？
   - 文件修改是否全部落地？有无漏改文件？
   - 步骤验证条件是否全部通过（编译/单测/联调）？
2. **回归检查**：
   - 项目全量构建是否通过？
   - 现有测试是否全部通过？
   - 用 `git status` / `git diff` 核对实际改动范围，确认未触及计划外的文件，多余改动需说明原因或还原。
   - 实际改动文件是否与「变更文件汇总」一致？联调测试矩阵的场景是否逐项验证？
3. **边界条件验证**（对照已实现代码）：
   - 空结果、空参数、并发冲突、重复操作（幂等）是否正确处理？
   - SQL/配置替换是否精确匹配源码（含原有缩进格式）？
   - 资源是否正确释放（连接/句柄/锁/事务），有无泄漏？
   - 常量/枚举值使用是否正确？
4. 输出复核报告（通过项 / 问题清单），逐项修正后重新确认。

### `wip-feishu-upload`

将 `WIP.md` 上传为飞书云文档，归类到 `work-in-process/{folderName}` 文件夹下，便于团队共享和评审。

1. 读取 `WIP.md`，确认文件存在且非空骨架（标题不为 `{需求标题}`）。
2. 读取 `.claude/skills/work-in-process/config.json` 获取飞书配置（`.claude/` 已被 `.gitignore` 忽略）：
   - `appId` — 飞书应用 App ID
   - `appSecret` — 飞书应用 App Secret
   - `folderName` — 子文件夹名称（可选，不填则直接放在 `work-in-process` 根目录下）

   配置模板：
   ```json
   {
     "appId": "cli_xxx",
     "appSecret": "xxx",
     "folderName": "crawl-data"
   }
   ```
   > 所有文档统一存放在 **云文档根目录 > `work-in-process` > `{folderName}`** 层级下。

3. 执行 `scripts/feishu_upload.py <WIP.md路径>`，流程：

   **步骤 A：获取 tenant_access_token**
   **步骤 B：查找或创建根目录 `work-in-process`**
   **步骤 C：在根目录下查找或创建子文件夹 `{folderName}`**（`folderName` 已配置时）
   **步骤 D：在目标文件夹中创建飞书文档**（标题 = `{日期}_{WIP标题}`）
   **步骤 E：写入 WIP.md 内容**（markdown → docx blocks，含标题层级）

4. 输出文档链接 `https://bytedance.feishu.cn/docx/{doc_id}`。

> **前置依赖**：`requests` 库（`pip install requests`），飞书应用需有 `docx:document` 和 `drive:drive` 权限。
>
> **已知限制**：当前仅支持标题层级（h1/h2/h3）和纯文本。表格、加粗、代码块、列表、水平线等 Markdown 元素会以原始文本呈现，不转换为 docx 格式。

### `wip-feishu-list`

列出飞书云空间中已上传的 WIP 文档。

1. 读取 `.claude/skills/work-in-process/config.json` 获取飞书配置。
2. 执行 `scripts/feishu_list.py [--tsv] [page_size] [page]`：
   - `--tsv`：可选，输出 Tab 分隔值（可直接复制粘贴到 Excel）
   - `page_size`：可选，每页数量（默认 100，最大 200）
   - `page`：可选，页码（默认第 1 页）
   - **步骤 A**：获取 `tenant_access_token`
   - **步骤 B**：查找或创建根目录 `work-in-process`
   - **步骤 C**：在根目录下查找或创建子文件夹 `{folderName}`，列出其中所有 `docx` 文档
3. 输出样式由 `templates/list.json` 控制：
   - `columns`：列定义（key / label / width），width=0 表示不截断
   - `separator`：分隔线字符
   - `dateFormat`：日期格式（如 `%Y-%m-%d`）
   模板缺失时使用内置默认样式。
4. 输出文档列表：序号、标题、日期、链接。

> **前置依赖**：`requests` 库（`pip install requests`），飞书应用需有 `drive:drive` 权限。

### `wip-feishu-delete`

删除飞书云空间中的 WIP 文档。

1. 读取 `.claude/skills/work-in-process/config.json` 获取飞书配置。
2. 执行 `scripts/feishu_delete.py <关键字|doc_id|--all>`：
   - **步骤 A**：获取 `tenant_access_token`，在 `work-in-process/{folderName}` 中列出文档
   - **步骤 B**：按参数匹配目标文档（标题关键字 / doc_id / token）
   - **步骤 C**：确认后调用 `DELETE /drive/v1/files/{token}` 移入回收站
3. 支持以下删除模式：
   - `wip-feishu-delete is_transfer` — 按标题关键字匹配（唯一匹配时直接删除，多项匹配提示用更精确的关键字）
   - `wip-feishu-delete FCjUdotqjoRpl0xcZqbcATO9nTd` — 按 doc_id 精确删除
   - `wip-feishu-delete --all` — 删除文件夹内全部 WIP 文档（需二次确认）

> **前置依赖**：`requests` 库（`pip install requests`），飞书应用需有 `drive:drive` 权限。

### `wip-feishu-search`

按关键字搜索已上传的 WIP 文档，输出匹配结果。

1. 读取 `.claude/skills/work-in-process/config.json` 获取飞书配置。
2. 执行 `scripts/feishu_search.py <关键字> [--tsv]`：
   - **步骤 A**：获取 `tenant_access_token`，在 `work-in-process/{folderName}` 中列出所有文档
   - **步骤 B**：按标题关键字过滤，输出匹配文档列表
3. 输出格式与 `wip-feishu-list` 一致（支持 `--tsv` 导出）。

> **前置依赖**：`requests` 库（`pip install requests`），飞书应用需有 `drive:drive` 权限。

### `wip-feishu-read`

按关键字搜索并阅读所有匹配的 WIP 文档完整内容。

1. 读取 `.claude/skills/work-in-process/config.json` 获取飞书配置。
2. 执行 `scripts/feishu_read.py <关键字|doc_id>`：
   - **步骤 A**：同 `wip-feishu-search`，在 `work-in-process/{folderName}` 中按标题关键字/doc_id 搜索匹配文档（支持多项匹配）
   - **步骤 B**：对每篇匹配文档，通过 `GET /docx/v1/documents/{doc_id}/blocks` 获取全部 block
   - **步骤 C**：将 blocks 渲染为 Markdown 风格文本输出（h1/h2/h3 + 正文），多篇文档间以分隔线区分
3. 输出每篇文档的标题、链接、完整内容，末尾汇总篇数。

> **前置依赖**：`requests` 库（`pip install requests`），飞书应用需有 `drive:drive` 和 `docx:document` 权限。

## 通用规则

### 每个步骤必须可执行

每个步骤用统一模板呈现，缺一不可：

- **文件**：目标文件路径，标注新建/修改/删除
- **位置**：精确到方法名 + 行号锚点（如 `CreatePerformance() 之后，第 68 行后`），让改动落点无歧义
- **背景**：为何改（仅在改动原因不直观时需要）
- **操作**：具体代码/SQL，可直接落地
- **验证**：编译/单测/联调的具体方式

禁止放入"仅确认""参考""梳理现有行为"等非执行项。

### 独立章节要求

以下为执行计划之外的独立顶层章节，必须单独成节：

- 「变更文件汇总」→ 独立章节，表格 `# | 文件 | 操作 | 对应阶段`，带阶段回溯列
- 「构建顺序」→ 独立章节，依赖图 + 阶段内并行链路说明

以下内容禁止独立成节，融入执行计划各步骤：

- 「数据库变更」→ 融入数据层相关阶段的步骤
- 「详细实现方案」→ 融入业务逻辑层相关阶段的步骤

### 自查清单

- [ ] 数据变更涉及事务时已正确处理原子性
- [ ] 常量/枚举定义完整，边界条件已正确处理
- [ ] SQL 或配置替换精确匹配源码（含原有缩进格式）
- [ ] 已规避目标语言的常见陷阱（如变量捕获/作用域、可变默认值、隐式类型转换等）
- [ ] 资源（连接/句柄/锁/事务）已正确释放，无泄漏
- [ ] 查询方法变更后空结果处理正确
- [ ] 新增字段/表/属性与现有命名风格一致
