# wip-build

生成设计文档（总体设计 + 模块设计），**自动判断模块数量**并根据职责**自动命名**。

## 执行流程

1. 读取当前项目代码结构，理解技术栈、命名风格
2. 与用户对话澄清需求（背景、范围、约束）
3. **自动判断模块数量**：
   - **简单需求**（单功能/单文件/单一逻辑）：单模块，自动命名（如 `core`, `validator`, `service`）
   - **复杂需求**（多独立子系统/多层架构）：自动拆分多个模块
4. **智能模块命名**（根据职责）：
   | 模块职责 | 自动命名示例 |
   |---------|-------------|
   | 数据模型定义 | `data-models` |
   | 数据库迁移 | `db-migrations` |
   | 业务逻辑实现 | `business-logic` |
   | API 接口层 | `api-endpoints` |
   | 权限校验 | `auth-validation` |
   | 缓存优化 | `cache-optimization` |
5. 生成分层设计文档：
   - **总体设计** → `.wip/{name}/design.md`
   - **模块设计** → `.wip/{name}/modules/{module}/design.md`
6. **自动更新 ledger.md**：标记设计完成，更新各模块状态

## 设计文档编写原则

**设计方案**面向临时评审，写一份总体设计概述：

- **总体思路**：用 3-4 句话讲清楚"要解决什么问题 → 用什么方式解决 → 核心设计原则"。不说具体表名、字段名、API 路径，让非技术读者也能看懂。
- **架构概要**：说明改动涉及的系统层次（数据层/业务层/接口层）、各层职责划分、模块间调用关系，标注涉及的数据表、接口、上下游依赖及是否向后兼容。
- **关键决策**：用表格列出 2-5 个关键设计取舍（如：为何选择该方案而非其他、状态机流转设计依据），每项包含方案/理由/风险。
- 整体控制在 **1 页篇幅**，语言精炼，便于快速评审决策。

**需求背景**写明：触发条件、范围边界、前置校验、状态约束、幂等语义。从对话中提取所有需求要点。

**变更文件汇总**作为独立章节，表格列出每个文件的操作类型（新增/修改/删除）及对应执行步骤，作为回溯速查表。

**构建顺序**用依赖图标注阶段间推进顺序及阶段内并行链路，点明关键步骤的前置依赖。

## 总体设计文档结构

- **参考资料**：业务流程图、数据表关系、关键方法签名
- **需求背景**：触发条件、范围边界、前置校验、状态约束、幂等语义
- **设计方案**：总体思路（3-4句）、架构概要（系统层次/职责划分/调用关系）、关键决策表（方案/理由/风险）
- **模块划分**：表格（模块/说明/前置依赖/状态）
- **变更文件汇总**：表格（文件/操作/所属模块）
- **依赖关系图**：模块间依赖顺序

## 模块设计文档结构

- **模块边界**：职责、不做的事、接口契约（输入/输出/错误处理）
- **数据模型**：新增或修改的实体、字段、类型
- **状态机**（如有）：状态流转图
- **实现思路**：技术选型、关键算法、设计模式
- **预估变更**：表格（文件/操作/说明）
- **依赖**：前置模块、后置模块

## 依赖

无外部脚本依赖。模块命名由 AI 根据职责直接推断，无需 Python 脚本辅助。
